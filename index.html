Machine Learning AI

In Skit Gubbe I've experimented with using Machine Learning to develop an AI opponent. I first made a simple opponent that only played the lowest card possible but later I started looking in to Q-Learning AI, which is a type of reinforcement learning where the AI learns by trial and error while playing itself. It plays the game over and over and gradually figures out which actions tend to lead to winning. It does that by saving states to a Q-Table which it then references while playing. If it has not seen that exact position/state before it will save it to the Q-Table and if its actions were good or bad which it decides through a reward system

Machine Learning Simulation

To teach the Q-Learning AI I used an empty scene in Unity to run millions of simulations where it plays itself and learns the states it then saved to the Q-Table. The training first starts with the AI playing randomly so it learns a lot of states early on. Later as the Epsilon value drops the less and less it plays randomly and instead refers to the states in the Q-Table, making its winrate go up throughout the simulation.

Tiny Train Tales

This project is directly inspired by the mobile game Tiny Rails and the intention of this game is purely educational. I had long been a fan of Tiny Rails and both wanted to see if I could recreate it and learn a lot from which I definitely have. This project helped me learn about A* path finding, JSON save files, inventory systems and how Game Objects relate to each other

JSON Save System

Before making this system I relied on Unity's PlayerPrefs function to store data, but PlayerPrefs is designed for simple key-value pairs like settings and not structured game data with nested objects. After adding Cargo and an Inventory System where each cargo object and city had many values to store, I felt it was necessary to add a custom Save System to make it both easier and more reliable. The system works by saving data into SaveData classes that are then stored inside SaveFile classes as lists. It creates a separate JSON file for each category of data like cities, inventory, upgrades and passengers, keeping things organized so only the relevant file needs to be rewritten when something changes. These files are placed locally on the player's computer. Cities are also stored in a dictionary keyed by city name so any system can look up a city instantly rather than scanning through the whole list. SaveToDisk() is called manually at key moments like scene transitions and game exit. When the player starts the game it reads the saved files and loads all the data back in.

Path Finding

This is the cornerstone of the entire game and the feature I've spent the most time on. The pathfinding uses the A* algorithm, where both the start city and destination are known in advance. It works by exploring cities one at a time, always prioritizing the one with the lowest combined cost. Each city tracks two values, the gCost, which is the actual distance traveled from the start to reach that city, and the hCost, which is an estimated distance remaining to the goal. These two are added together into an fCost, and the city with the lowest fCost is always explored next. This means paths that are expensive and still from the goal doesn't get prioritized, while cheaper and more promising routes get explored first. Each city also keeps track of which city it came from, so once the destination is reached the full path can be retraced backwards from end to start. The result is the shortest valid route between any two cities on the map.
